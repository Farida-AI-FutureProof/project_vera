# Dataset Mapping

This document outlines how to map fields from publicly available datasets into the unified `ChromaDB` ingestion format used by Project VERA. Each source dataset has its own schema; by mapping the fields consistently, we enable cross-domain retrieval, discrepancy detection and RBAC enforcement.

## Aviation Accident Data (NTSB)

**Source**: National Transportation Safety Board aviation accident database (Kaggle). Each row describes an accident or incident.

| Dataset Field | Description | Mapped Metadata |
|---------------|-------------|-----------------|
| `EventId` | Unique identifier for the accident/incident | `document_id` |
| `Investigation_Type` | `Accident` or `Incident` | `source` = `aviation` |
| `Accident_Number` | Human‑readable reference number | part of `document_id` |
| `Event_Date` | Date of the occurrence | store in `metadata['date']` |
| `Location`, `Country` | Free‑text location details | `department` = `operations` (example) |
| `Narrative` | Long textual description of the event | Document text for RAG |
| `Category`, `Injury_Severity`, ... | Additional features | can be included in document body for context |
| `access_level` | not present in dataset | assign `public` for general accident reports |

When ingesting, treat each accident narrative as a document chunk. Tag the document with `source='aviation'`, `document_id` equal to `EventId`, and an appropriate `access_level` (most reports are public). Optionally assign a `department` value (e.g., `operations`) to reflect the functional area.

## FDA Enforcement / Recall Data

**Source**: FDA Enforcement Reports dataset. Each row describes a recall.

| Dataset Field | Description | Mapped Metadata |
|---------------|-------------|-----------------|
| `Recall_Number` | Unique recall identifier | `document_id` |
| `Classification` | Class I/II/III | Include in document text |
| `Product_Description` | Description of the product | part of document text |
| `Reason_for_Recall` | Why the recall was issued | part of document text |
| `Recall_Initiation_Date`, `Report_Date` | Timestamps | store as `metadata['date']` |
| `State`, `Country` | Distribution geography | `department` = `supply_chain` or `quality` |
| `access_level` | not present | assign `public` for published enforcement reports |

For recalls, create a document per recall entry. Combine fields such as `Product_Description` and `Reason_for_Recall` into the document body. Use `source='fda'`, `document_id=Recall_Number`, and assign an `access_level` (usually `public`). Set `department` to the function responsible for the product (e.g., `quality` or `supply_chain`).

## Supply Chain / Logistics Data

**Source**: Supply chain datasets capturing shipments, delivery schedules and operational metrics.

| Dataset Field | Description | Mapped Metadata |
|---------------|-------------|-----------------|
| `OrderID` / `Shipment_ID` | Unique identifier for a shipment/order | `document_id` |
| `Route` / `Destination` | Delivery path or destination | `department='logistics'` |
| `Planned_Delivery_Date` | Expected delivery time | part of metadata (`date`) |
| `Actual_Delivery_Date` | Actual completion time | document body or metadata |
| `Status` | Delivered, In Transit, Delayed | include in document text |
| `SLA_Hours` / `Delay` | Service level agreement and actual delay | part of document for discrepancy checks |
| `access_level` | not present | assign `internal_only` for internal logistics records |

When ingesting logistics data, treat each shipment record as a document. Compose a narrative description summarizing the shipment and its status. Use `source='logistics'`, `document_id` equal to the shipment or order identifier, and `access_level='internal_only'`. Assign `department='logistics'`.

## Synthetic Emails and Change Logs

Synthetic emails and change approvals simulate informal communications. These documents are created manually or generated by the team.

| Field | Description | Mapped Metadata |
|-------|-------------|-----------------|
| `email_id` | Unique identifier for the email | `document_id` |
| `subject`, `body` | Email content | Document text for RAG |
| `sender_role` | Role of the sender (e.g., Senior Engineer) | may inform `access_level` |
| `timestamp` | Time the email was sent | `metadata['date']` |
| `access_level` | Confidentiality level | `internal_only` or `confidential` |
| `department` | Which team the email belongs to | assign relevant department (e.g., `engineering`, `quality`) |

Tag each synthetic email with `source='email'`. Use appropriate `access_level` (`internal_only` or `confidential`) and `department` to simulate RBAC.

## Access Levels and Roles

VERA uses three access levels: `public`, `internal_only`, and `confidential`. When mapping datasets:

- **Public**: Documents accessible to all roles (e.g., published accident reports, enforcement notices).
- **Internal Only**: Operational records and internal emails that junior roles cannot access.
- **Confidential**: Sensitive communications or documents requiring senior or compliance permissions.

Ensure every document has a defined `access_level` and `document_id` for retrieval and RBAC enforcement.

For additional details on metadata fields and their definitions, refer to `ingestion/metadata_schema.md`.
